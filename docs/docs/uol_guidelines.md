# University of Liverpool Guidelines

## Design

Although you only need to submit the design documentation at the end of term, you will effectively take some design decisions earlier on. You may need to decide, for instance,
- How to structure your data(base)
- How to organize your interface
- Generally, how the system will work

You will be asked to provide some details about this in your proposal (requirement document) but it is fair to assume that further decisions will be needed later.

---

### Design Document

Although designs will vary according to the needs of particular projects, a typical design document will comprise
- a description of the system's components and how they are organised (system's architecture)
- a description of the data used within the system
- a description of the processes and algorithms used to manipulate the data
- a description of the system's interface(s)
- a design of the evaluation process.

It is up to each team to decide how to organise and describe this information, but obviously, like for any written document, planning its structure is important. You may want to consult the Writing page to get some advice or inspiration on what you should do, and what you should NOT do in relation to writing style and the general structure of a report.

Regarding the technical content of the design document, if the development follows an object-oriented methodology, one might use use-case diagrams, interaction charts (aka event traces), class diagrams, pseudo-code descriptions of the key processes, state charts...

If the system is data-centred and a more traditional methodology is preferred, Connolly and Begg's book can be used as a source of inspiration. Data dictionaries, system boundary diagrams, entity-relationship (ER) diagrams, table structures, and transaction matrices are all valid tools to describe different system aspects.

For a project involving the empirical investigation of some hypothesis one would normally expect to see in addition things such as a statement of the hypotheses to be tested, a description of the test data to be used, an experiment design, the experiments to be performed, any control to be used, a description of how the results will be analysed, including any statistical techniques that will be used, a brief description of the anticipated conclusions. 

For a project attempting to devise new algorithms one would normally expect to see in addition things such as a clear description of the problem to be solved, a description of the existing algorithms of related problem and a critical evaluation of them (e.g., why they are not applicable in your project), a description of the approach used to solve the problem, a description of how the new algorithms will be analysed, including mathematical and experimental analysis. For the mathematical analysis to be carried out, a description of the mathematical model to be used, a description of the performance metric by which the algorithm will be measured, ranging from correctness, running time, optimality or approximability for some objective functions, a description of the control against which your algorithm is compared, e.g., the optimal algorithm, are all useful. For the experimental analysis to be carried out, an experiment design, the experiments to be performed, any control to be used, a description of how the results will be analysed, including any statistical techniques that will be used, might be included.

Alternative techniques, such as data flow diagrams, navigation path diagrams, storyboards, functional descriptions or components, can also be used.

YOU DO NOT NEED TO INCLUDE ALL THE DOCUMENTS OR DIAGRAMS MENTIONED ABOVE!! The important thing is that the documentation clearly shows that a design methodology has been followed, and that the design has been carried out with sufficient attention to detail to suggest that no major requirement has been missed, and the resulting system fulfils its purpose.
